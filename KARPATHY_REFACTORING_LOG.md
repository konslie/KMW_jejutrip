# Karpathy 가이드라인 기반 리팩토링 및 버그 수정 로그

## 적용된 기준 (Karpathy Guidelines)
이번 리팩토링 및 버그 수정은 다음 두 가지 핵심 가이드라인에 따라 진행되었습니다.

### 1. Simplicity First (단순함 우선)
* **원칙:** "문제를 해결하는 최소한의 코드. 추측하지 마라. 요청받지 않은 기능이나 불필요한 추상화를 추가하지 마라." 
* **적용:** 애플리케이션의 핵심 목적에 벗어나는 기능이나, 불필요하게 깊은 함수 호출 스택 스파게티(Call Stack)를 구성하던 요소를 찾아내어 단순화했습니다.

### 2. Surgical Changes (수술적 변경)
* **원칙:** "필요한 부분만 건드려라. 목적 중심의 실행."
* **적용:** 기존의 동작은 최대한 유지하면서, 발견된 명백한 버그만을 정확하게 타겟팅하여 수정하고, 삭제된 변경사항으로 인해 고아가 된 코드(Dead Code)를 깨끗하게 정리했습니다.

---

## 주요 작업 내용

### 1. "관리자용 이스터에그" 로직 최적화 (단순화 적용)
당초 이스터에그 기능 자체를 삭제하는 것을 제안했으나, 사용자 요청에 따라 기능은 유지하되 **오버엔지니어링된 구조를 단순화**하는 방향으로 리팩토링했습니다.
* **관련 파일:** `src/main.js`, `src/data/schedule.js`, `src/api/firebase.js`
* **작업 사항:**
  * `schedule.js`에 존재하던 쓸데없는 중간 래퍼(Wrapper) 함수들(`triggerAdminRestore`, `triggerAdminSave`)을 제거했습니다.
  * `main.js`에서 Firebase API를 직접 호출하도록 흐름을 직관적으로 변경하여 모듈 간의 결합도를 낮추고 추상화를 줄였습니다.
  * `firebase.js` 내부의 복잡한 에러 처리 블록이나 불필요한 내부 상태 플래그(`isInternalUpdate`) 사용을 완화하여 코드를 간결하게 만들었습니다.

### 2. 검색 UI 하드코딩 버그 수정 (수술적 변경 적용)
장소 검색 후 일차를 선택할 때, 선택지가 4일차까지만 고정되어 있던 하드코딩 버그를 동적 환경에 맞게 수정했습니다.
* **관련 파일:** `src/ui/search.js`
* **작업 사항:**
  * `<select>` 태그의 옵션 목록을 `day1` ~ `day4`로 하드코딩된 문자열에서 동적 렌더링으로 변경했습니다.
  * `config.js`를 통해 로드된 동적 데이터(`config.totalDays`)를 기반으로 사용자의 설정 일수에 맞춰 선택 옵션이 1일차부터 N일차까지 정확히 나타나도록 수정했습니다.

### 3. 설정 파일 로드 경로 수정 (동작 정상화)
* **관련 파일:** `src/data/config.js`
* **작업 사항:**
  * 로컬 환경에서 JSON 데이터가 바인딩되지 못하는 문제를 해결하기 위해, `fetch` 경로를 `./trips/`에서 올바른 디렉터리 구조인 `./public/trips/`로 수정하여 동적 데이터를 제대로 받아오게끔 조치했습니다.
